Project 1: AI-Powered Conversational Assistant with Tool Integration, Persistent Memory & Human-in-the-Loop Moderation
A full-stack conversational AI application built with LangGraph and Streamlit that combines real-time tool use, persistent multi-session memory, human-in-the-loop approval gates, and production observability.

The backend orchestrates a ReAct-style agent loop where the model autonomously decides whether to respond directly or invoke tools — including live web search (DuckDuckGo), arithmetic calculation, and real-time stock price retrieval (Alpha Vantage API). A human-in-the-loop (HITL) moderation layer is built directly into the graph using LangGraph's interrupt primitive — before the model generates any response, execution pauses and surfaces the pending question to a human moderator for approval. If approved, the model proceeds; if rejected, a controlled fallback response is returned
Key capabilities:
•	ReAct-style agent loop with autonomous tool selection and invocation
•	Live web search (DuckDuckGo), arithmetic calculator, and real-time stock prices (Alpha Vantage)
•	Human-in-the-loop approval gate using LangGraph interrupt — execution pauses for moderator sign-off before any response is sent
•	SQLite-backed persistent memory with thread isolation for multi-session support
•	Streamlit frontend with ChatGPT-style conversation sidebar, token-level streaming, and inline moderation panel
•	LangSmith tracing per thread for full production observability

Tech Stack: LangGraph · LangChain · Streamlit · HuggingFace (Qwen2.5-7B) · SQLite · SqliteSaver · MemorySaver · DuckDuckGoSearchRun · Alpha Vantage API · LangSmith · ToolNode · interrupt · Command · Python
Project 2: Corrective RAG (CRAG) Pipeline with Hybrid Retrieval and LLM-Based Context Refinement
An advanced RAG system built with LangGraph that goes beyond naive retrieval by introducing a scoring-based quality gate on retrieved documents before any answer is generated. Rather than blindly passing retrieved chunks to the LLM, the pipeline evaluates each chunk individually and routes the query through one of three paths based on confidence — making it significantly more reliable than standard RAG in production settings.

Key capabilities:
•	Per-chunk LLM relevance scoring (0.0–1.0) against the user question before generation
•	Three-way routing: CORRECT (internal docs only) / INCORRECT (web search fallback) / AMBIGUOUS (internal + web combined)
•	LLM query rewriting before Tavily web search to improve retrieval signal
•	Sentence-level decomposition and LLM relevance filter — only sentences that directly help answer the question are passed to the generator
•	Final answer generation using only the refined, filtered context

Tech Stack: LangGraph · FAISS / Chroma DB· HuggingFaceEmbeddings · HuggingFaceEndpoint · ChatHuggingFace (Qwen2.5-7B) · TavilySearchResults · PyPDFLoader · Pydantic · ChatPromptTemplate · Python

Project 4: Multi-Tier Memory Architecture for Personalised AI Assistants
A series of progressively advanced conversational AI memory systems built with LangGraph, exploring three distinct memory tiers — in-process, database-backed short-term, and persistent long-term — demonstrating a complete understanding of how production AI assistants manage context across turns, sessions, and application restarts.

Tier 1 — Token-Aware Short-Term Memory (InMemorySaver)
A conversational agent with token-budget-aware message trimming that automatically prunes older messages when the context window approaches its limit. Ensures cost-controlled, coherent conversations without hitting model token limits in long sessions.

Tier 2 — Persistent Multi-Session Memory (PostgresSaver)
A multi-user chatbot with conversation history persisted to PostgreSQL via LangGraph's checkpointer, with each conversation isolated by thread ID. Users can resume any previous conversation exactly where they left off, surviving application restarts and redeployments.

Tier 3 — Long-Term User Memory with Deduplication (InMemoryStore to PostgresStore)
A personalised AI assistant that extracts and stores atomic user facts (name, preferences, projects, goals) across sessions using a dedicated memory extraction LLM. A second LLM judge decides whether each new piece of information is genuinely new before writing it, preventing duplicate memories. The assistant uses stored memories to personalise every response — addressing users by name, referencing their tools and projects, and suggesting contextually relevant follow-up questions. Built first with InMemoryStore then upgraded to PostgresStore for full cross-session persistence verified across kernel restarts.

Tech Stack: LangGraph · PostgresSaver · SqliteSaver · InMemorySaver · PostgresStore · InMemoryStore · MessagesState · trim_messages · count_tokens_approximately · ChatHuggingFace (Qwen2.5-7B + GPT-OSS-20B) · Pydantic · PostgreSQL · SQLite · Python

Project 5: Parallel Multi-Agent Blog Writing Pipeline
An agentic content generation system built with LangGraph that takes a topic and autonomously plans, researches, and writes a complete technical blog post — with multiple specialised agents working in parallel to produce each section simultaneously. Blog content saved as markdown file. 

Key capabilities:
•	Router classifies topic into closed-book, hybrid, or open-book research mode
•	Research agent runs multiple Tavily web searches, deduplicates by URL, and packages evidence into structured EvidenceItem objects
•	Orchestrator agent produces a structured blog plan with 5-9 sections, each with goal, bullets, target word count, and flags for code/citations/research
•	LangGraph Send API fans out each section to parallel worker agents, all writing simultaneously
•	Reducer node collects and reorders completed sections by ID before assembling final Markdown output
•	Grounding policies prevent hallucination in research-heavy topics — open-book mode requires every claim to be backed by an evidence URL

Tech Stack: LangGraph · Send API · TavilySearchResults · ChatHuggingFace (Qwen2.5-7B) · HuggingFaceEndpoint · Pydantic · ChatPromptTemplate · LangSmith · Python
Business Value: Demonstrates an AI content pipeline that can reduce blog post production time from hours to minutes, with built-in grounding policies that prevent the model from fabricating facts in research-heavy topics. Project 6: Multi-Server MCP Agent with Async Tool Orchestration
A production-oriented agentic system that connects a single LLM to multiple external services simultaneously using the Model Context Protocol (MCP) — an emerging open standard for connecting AI models to real-world tools and data sources.

Key capabilities:
•	Connects concurrently to two MCP servers with different transport types: local stdio (FastMCP Python server) and remote HTTP (ExpenseTracker) with bearer token auth
•	Tools discovered dynamically at runtime and bound to the model
•	Full async tool-calling loop: model selects tool, arguments extracted, MCP tool invoked via ainvoke, result injected back for final grounded response
•	Additional tools include real-time web search and live weather API
•	Demonstrates secure multi-server tool exposure across both local and cloud-hosted MCP endpoints

Tech Stack: langchain_mcp_adapters · MultiServerMCPClient · FastMCP · stdio transport · streamable_http transport · asyncio · ChatHuggingFace (Qwen2.5-7B) · ToolMessage · LangChain · Python
